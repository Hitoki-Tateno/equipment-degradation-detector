# ADR: 表示層からのデバッグ用データ操作（CSVインポート + 一括削除）

- 日付: 2026-03-03
- ステータス: 承認済み
- 関連ADR: architecture_minutes.md, tech_selection_minutes.md

## 背景

### 現行アーキテクチャの前提

architecture_minutes.md で定義されたデータフローでは、データ取り込みはコンテナ外のアダプター経由で行われる:

```
[コンテナ外]                     [コンテナ内]
  RDB --> アダプター --> 取り込みAPI --> Store層
  CSV --> アダプター --/
  JSON -> アダプター -/
```

表示層（React）は読み取り専用として設計されている:

- Store層から生データを読み取り（境界②）
- 結果ストアから分析結果を読み取り（境界③）
- モデル定義の保存・削除（PUT/DELETE `/api/models`）は表示層の操作だが、結果ストアへの書き込みであり、データ取り込み層への書き込みではない

### 課題

開発・デバッグ時にCSVファイルのインポートやストアの全削除をcurlやSwagger UIで行う必要があり、非効率。バックエンドAPIは既に存在する（`POST /api/records/csv`, `DELETE /api/debug/*`）が、GUIからのアクセス手段がない。

## 決定事項

### 決定: 表示層にデバッグ専用のデータ操作UIを設置する

**方針**: フロントエンドにデバッグ設定ページを追加し、以下の操作をGUIから実行可能にする:

1. **CSVインポート**: 表示層 → 取り込みAPI（`POST /api/records/csv`）
2. **データ一括削除**: 表示層 → デバッグAPI（`DELETE /api/debug/*`）

**データフロー変更**:

```
[コンテナ外]                     [コンテナ内]

  RDB --> アダプター --> 取り込みAPI --> Store層(SQLite)
  CSV --> アダプター --/                  |     |
  JSON -> アダプター -/                分析層  | データ提供API
                                  |        |
                                  v        v
                              結果ストア --> データ提供API --> 表示層(React)
                                  |                          ↑  |
                              EventBus -----> SSE ----------/  |
                                                               |
  [デバッグ専用]  表示層 --> 取り込みAPI (CSVインポート)         |
                  表示層 --> デバッグAPI (一括削除) -------------/
```

表示層から取り込みAPIへの書き込みパスが新設される。これは既存のアダプターパターン（コンテナ外→取り込みAPI）とは別のデバッグ専用経路。

### スコープと制約

| 項目 | 決定 |
|------|------|
| 用途 | 開発・デバッグ専用。本番運用のデータ投入経路ではない |
| 表示層の責務変更 | 既存のモデル定義操作（PUT/DELETE `/api/models`）と同様、APIを経由した間接操作。Store層への直接依存は発生しない |
| バックエンド変更 | なし。既存APIをそのまま使用 |
| 層間依存ルール | 違反なし。フロントエンドはHTTP API経由でのみアクセスし、バックエンドの `interfaces/` → `store/` の依存ルールに影響しない |
| 着脱性 | コンポーネントは props なし・コールバックなしの自己完結型。App.js から3行（import + メニュー + 条件レンダリング）を削除するだけで除去可能 |
| SSE連携 | CSVインポートはバックエンドで `dashboard-updated` イベントを発火する（既存動作）。デバッグ削除は発火しない（意図的） |

### 本番運用時の扱い

デバッグ設定ページはフロントエンドのみの変更で着脱可能なため、本番リリース時にメニューから除外する選択肢がある。バックエンドのデバッグエンドポイント（`DELETE /api/debug/*`）の本番公開可否は別途判断する。

**理由**:

- 既存のバックエンドAPIをそのまま活用でき、新規エンドポイントの追加が不要
- フロントエンドの疎結合設計により、着脱コストがほぼゼロ
- 開発効率の向上（curl不要、結果を即座にダッシュボードで確認可能）
- アダプターパターンの本来の目的（外部データソースの抽象化）は維持される。GUIからのCSVインポートはユーザーが手動でファイルを選択する開発支援であり、自動化されたデータパイプラインとは別の経路

## 影響範囲

### フロントエンドのみ（バックエンド変更なし）

- `frontend/src/services/api.js` — API関数4つ追加（uploadCsv, deleteDebugData, deleteDebugResults, deleteDebugAll）
- `frontend/src/components/DebugSettings.jsx` — 新規コンポーネント
- `frontend/src/App.js` — メニュー項目 + 条件レンダリング追加
- `frontend/src/App.css` — メニュー幅調整

### 依存ルール

バックエンドの依存ルール（`test_dependency_rules.py`）に影響なし。フロントエンドはHTTP APIの消費者であり、Pythonモジュールの依存グラフに含まれない。
